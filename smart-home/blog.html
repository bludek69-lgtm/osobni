2026-05-22 FIX

Roleta jídelna zavřela přes otevřené okno (P1) — silent-fail kvůli rename device

Předchozí incident stejného typu, jiný root cause. sh_jidelna_roleta_router_v2 měl v lokálním D constu hardcoded starý device name 'Door/Window Sensor (SNZB-04)1', ale device byl mezitím přejmenován na 'Door/Window Sensor Jidelna1'. getDev() vrátil null, try/catch(_) ho potichu spolkl, windowOpen defaultoval na false (unsafe default) → SAFETY skip se nespustil → roleta zavřela přes otevřené okno. Fix commit 349532d: dva string replaces (oba název senzoru), commit + push. Pravidla z incidentu: (1) safety guards musí default-to-fail-CLOSED (windowOpen=true při fetch failu), (2) catch bloky musí volat logEvent aby silent fail šel vidět, (3) device names číst ze sh_device_map namísto hardcoded stringů. 2026-05-19 FIX

SIGABRT každých 5 minut — group concurrency, ne vadný script

Čtyři dny audit memory, knihoven a logování. Nasazený forensic patch na 80 hot-path scriptech (odstranění Promise.race + wait() z inline fetchWithTimeout). Crash cyklus pokračoval. Teprve když uživatel vizuálně oznámil „padá to přesně každých 5 minut", příčina začala konvergovat. Nešlo o jeden vadný script — bylo to skupinová souběžná zátěž: 17 Advanced Flow s triggerem cron:every_nth {n:5, type:minute} spouštělo HomeyScript synchronně na 00s 5min hranici, runtime nezvládl souběh. Emergency 5-min isolation (vypnutí celé /5min HomeyScript skupiny) zastavila pády — crashedCount +0 za 12 minut. Pak R1 obnovil 3 kritické flows s delay 30/60/90s (Heating, Presence Failsafe, Brain Guardian). R1B rozprostřel 8 lehkých flows přes delays 5–195s. R1C snížil kadenci 3 nejtěžších na /10m (Device Availability, Health Monitor, OpenSpace Autoheal) — mem peak spadl z 235 MB na ~145 MB median (−38 %). ENABLE-ALL sequential vrátil zbývajících 6 heavy flows (Dashboard, Health Alert, Context Full, Diagnostic, Meta, AI Guard) na /15m nebo /30m s offsetem 215–320s, 1 PUT na flow + reálné fire ověření + 10 min observační okno. Tři hodiny, 6 kroků, 0 crashes během deploye. Žádný HomeyScript PUT, žádná business logika dotčená, žádný manual restart. Klíčová lekce: před hlubší crash analýzou nejdřív spočítej intervaly mezi pády — pravidelný interval = trigger-driven cause.

2026-05-18 INFRASTRUCTURE

SSD logging cutover — 1 var flip migroval 91+ logEvent callerů

2026-05-18 v 07:09 CEST proběhl migrační moment, na který se v Homey velmi obtížně reaguje retroaktivně. Až do tohoto okamžiku všechny logEvent(...) hovory napříč 91+ skripty směřovaly na Google Sheets Apps Script web app — pomalý, rate-limited a závislý na cloudu. Nasazená architektura umístila SQLite receiver na HA RPi (http://192.168.1.xxx:9080/log/sheets-compat) s daily JSONL files + WAL-enabled SQLite (4.2 MB po prvním dni). Migrace bez touching jediného skriptu: stačila změna jediné Logic proměnné sh_sheets_url z Apps Script URL na receiver URL. Skripty si target endpoint čtou per-call přes getValue('sh_sheets_url'), takže všechny callery se přepojily okamžitě. Google Sheets endpoint zůstává jako záložní fallback (revert = swap původní URL zpět). Schema kompatibilita 100 % (payload format sheets-compat), zero data loss. Po cutover proběhla séria operací na vrácení Energy Dashboard R1 aggregatoru + UI a osobní web AI page deploye.
